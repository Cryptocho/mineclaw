# MineClaw 项目计划

## 项目概述

**MineClaw** 是一个革命性的 AI 智能体平台，从传统的单线程、人工切换上下文的 Claw 架构，演进到**多 Agent 并行 + 自动化编排协作**的新一代架构。

### 核心愿景

- **从"单线程"到"并行协作"**：不再是一个 Agent 顺序执行任务，而是多个 specialized Agent 同时工作
- **从"人工切换"到"自动化编排"**：由系统智能地分解任务、分配工作、协调协作，无需人工干预
- **从"单一能力"到"专业分工"**：不同 Agent 具备不同专长（代码专家、文档专家、测试专家等），协同完成复杂任务
- **从"被动执行"到"主动规划"**：系统能自主理解目标、制定计划、监控进度、调整策略

### 为什么需要多 Agent 架构？

传统 Claw 的局限性：
1. **单线程瓶颈**：一个 Agent 同时只能做一件事，效率低下
2. **注意力分散**：同一个 Agent 需要在不同任务间切换，质量下降
3. **能力局限**：单个模型难以在所有领域都达到专家水平
4. **人工依赖**：需要用户不断引导和切换上下文

MineClaw 的突破：
1. **并行执行**：多个 Agent 同时工作，显著提升效率
2. **专业分工**：每个 Agent 专注特定领域，质量更高
3. **智能编排**：系统自动协调，无需人工干预
4. **弹性扩展**：可根据需要动态增加或减少 Agent

## 核心架构设计

### 整体协作模式

MineClaw 采用**混合协作模式**，结合两种核心协作方式：

#### 1. 集群并行模式
多个 Agent 同时工作，各自承担不同子任务，适用于：
- **创意脑洞**：多个 Agent 从不同角度发散思维
- **API 查询与学习**：并行查询文档源、交叉验证
- **大项目代码分工**：约定接口后，每个 Agent 实现一部分
- **多进程测试**：并行执行测试用例，大幅缩短时间

#### 2. 接力转交模式
任务被拆分为多个阶段，每个阶段由专门的 Agent 完成，完成后通过 JSON 工单转交下一 Agent，适用于：
- **持续犯错**：当前 Agent 陷入困境时，换新鲜视角
- **复杂项目分阶段**：每个阶段刷新上下文，避免过长

### 层级化总控架构

```
主总控 (Master Orchestrator)
    │
    ├── 分支总控 1 (Branch Orchestrator)
    │   ├── Agent A
    │   ├── Agent B
    │   └── Agent C
    │
    ├── 分支总控 2 (Branch Orchestrator)
    │   ├── Agent D
    │   └── Agent E
    │
    └── ...
```

#### 总控 Agent 的特殊能力
- **最强模型配置**：通常使用能力最强的模型
- **创建 Agent 实例**：可以编写提示词动态创建新 Agent
- **编写工作流**：定义任务分解和执行流程
- **计划与评审**：只在每一小步的开始和结束时介入
- **可被重置**：总控犯错也会被上下文管理 Agent 识别并重置

#### 汇报机制
- **分支总控**：处理分支内的反馈
- **主总控**：只在分支结束时介入
- **汇报内容**：只包含完成内容和未解决的问题

### 上下文管理 Agent (Context Manager)

上下文管理 Agent 是整个工作流的核心，具有以下特点：

#### 核心职责
1. **上下文裁剪**：像修改文件一样，运用修改工具裁剪和总结上下文
2. **转交判断**：决定是否需要将任务转交给下一 Agent
3. **路由再判断**：在上下文达到上限被裁剪后，重新判断路由策略
4. **质量监控**：监控包括总控在内的所有 Agent 的表现

#### 设计原则
- **轻量配置**：使用轻量模型，保持响应速度
- **相同上下文长度**：与其他 Agent 看到相同的信息，保持公平性
- **JSON 格式输入**：上下文以结构化格式提供，便于处理

#### 裁剪策略参考（提示词模板）
- **持续犯错部分**：裁剪掉反复失败的尝试
- **Bug 解决过程**：保留结果，裁剪过程（未来用于经验学习）
- **终端长输出**：裁剪冗余，如测试日志只保留测试成功/失败总结
- **冗余对话**：删除重复、无关的内容

#### 裁剪质量评估
- 裁剪提示词会被记录
- 由评审 Agent 或用户根据任务完成质量评分
- 评分结果将参与未来的提示词自动优化（Phase 7+）

### 工单机制 (Work Order)

接力转交时使用 JSON 格式工单，包含：

```
工单内容规范：
├── 已完成部分详细标注
├── 相关文件的相对路径列表
└── 下一阶段的详细计划
```

**设计原则**：工单应简洁、聚焦，仅包含必要信息，避免成为新的上下文负担。

### 路由模型 (Router Model)

#### 触发时机
1. **任务开始时**：初始判断使用单 Agent 还是多 Agent 模式
2. **上下文裁剪后**：上下文被管理 Agent 裁剪后，重新评估路由策略

#### 路由决策逻辑

**简单任务** → 单总控 Agent 模式

**复杂任务** → 判断集群还是接力模式

**集群模式适用场景**：
✅ 创意脑洞 - 需要多元视角
✅ API 查询与学习 - 需要并行查询和验证
✅ 大项目代码分工 - 需要约定接口后并行实现
✅ 多进程测试 - 需要并行执行测试

**接力模式适用场景**：
🔄 持续犯错 - 需要新鲜视角
🔄 复杂项目分阶段 - 需要刷新上下文，保持专注

**混合模式**：一个项目可以同时使用两种模式
- 集群为每个 Agent 分派任务
- 每个 Agent 自己把任务拆分成 Baby Steps，通过接力转交完成
- 每个集群分支都有独立的分支总控管理

### 工具掩码策略

#### 掩码范围
- **主要对象**：MCP 工具（而非本地工具）
- **掩码单位**：单个工具（而非整个 MCP 服务器）

#### 特殊工具
- **终端工具**：所有 Agent 都可以使用，无限制

#### 分配策略
- 总控在分配任务时为 Agent 选择可用工具
- 确保分配的工具通常足够完成任务
- 避免工具过多导致的分心

### Checkpoint 与会话管理

- **强关联设计**：Checkpoint 与 Session 一一对应
- **生命周期管理**：
  - Session 归档 → 相关 Checkpoint 归档或删除
  - 由 AgentFS 提供完整实现
- **轻量设计**：完全不会成为系统负担

### 转交预热 (Handover Warm-up)

**必须流程**：在正式转交任务前，需要进行预热

- 让新 Agent 先理解工单内容
- 确认关键信息无遗漏
- 准备好后再正式接管

### 单向流水线设计

- **设计原则**：流水线是单向的，不支持回传
- **消息追踪**：每个 Agent 的回复都有唯一的消息 ID 记录
- **简化协作**：避免复杂的双向依赖，让流程更清晰可控

### IDE 协同：Agent Client Protocol (ACP) 集成

MineClaw 将通过 **Agent Client Protocol (ACP)** 与用户习惯的 IDE 无缝协同工作。

#### 什么是 ACP？
- **类似 LSP 的标准化协议**：专门用于标准化 IDE/编辑器与 AI Agent 之间的通信
- **MCP 友好**：构建在 JSON-RPC 之上，尽可能重用 MCP 类型
- **本地 + 远程支持**：
  - **本地 Agent**：作为编辑器子进程运行，通过 stdio 的 JSON-RPC 通信
  - **远程 Agent**：可在云端或独立基础设施上托管，通过 HTTP 或 WebSocket 通信
- **主流 IDE 支持**：已被 Zed、JetBrains 等支持

#### ACP 集成价值
- **随时随地派任务**：用户可以在 IDE 中直接与 MineClaw 交互
- **无缝体验**：Agent 可以直接访问编辑器中的文件、终端、MCP 服务器
- **生态兼容**：符合 ACP 标准的 MineClaw 可以在所有支持 ACP 的编辑器中使用
- **双向通信**：
  - Agent 可以向编辑器请求工具调用权限
  - 编辑器可以向 Agent 推送实时更新（通过 JSON-RPC 通知）

#### 集成策略
- **官方 Rust 库支持**：使用 `agent-client-protocol` crate
- **双重访问模式**：
  1. **REST API + Flutter 前端**（传统 Web 方式）
  2. **ACP 协议**（IDE 协同方式）
- **共享核心逻辑**：两种访问方式共享相同的多 Agent 编排核心

## 关键设计决策回顾

### 1. 混合协作模式
- 结合集群并行和接力转交两种模式的优势
- 一个项目可以灵活组合使用两种模式

### 2. 层级化总控
- 主总控 + 分支总控的多层架构
- 总控只在关键节点介入，避免过度干预
- 总控也可以被重置，避免单点故障

### 3. 上下文管理 Agent 是核心
- 轻量但关键的角色
- 负责裁剪、转交判断、质量监控
- 使用与其他 Agent 相同的上下文长度

### 4. 工单机制简洁聚焦
- 只包含必要信息：已完成、文件路径、下一步计划
- 避免成为新的上下文负担

### 5. 工具掩码精细控制
- 按单个工具粒度掩码
- 终端工具全开放
- 总控负责为 Agent 分配合适的工具

### 6. Checkpoint 与会话强关联
- 简化管理，避免孤立 Checkpoint
- 由 AgentFS 提供支持，轻量高效

### 7. 转交预热必须
- 确保平滑过渡
- 避免信息丢失

### 8. 单向流水线
- 简化协作逻辑
- 避免复杂的双向依赖
- 完整的消息 ID 追踪

### 9. 裁剪质量可评估
- 为未来的自动优化积累数据
- 支持人工和自动评审

### 10. 为经验学习预留接口
- Phase 6 就开始为 Phase 7 准备数据结构
- 确保平滑过渡到持续进化阶段

### 11. ACP 协议优先
- 与 IDE 协同是核心能力
- ACP 和 Web API 共享核心逻辑
- 支持本地和远程两种部署模式

## 开发原则

1. **Baby Steps™**: 每次只做一个小的、有意义的改变
2. **The Process is the Product**: 过程和结果同样重要
3. **Incremental Validation**: 每一步都要验证
4. **Documentation First**: 先写文档再实现
5. **Concurrency Safety**: 并发安全是首要考虑
6. **Fault Tolerance**: 设计要考虑容错和恢复
7. **单向流程**: 保持流水线单向，简化协作
8. **可观测性**: 所有关键操作都要可追踪、可审计
