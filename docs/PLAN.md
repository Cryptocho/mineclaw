# MineClaw 项目计划

## 项目概述

**MineClaw** 是一个革命性的 AI 智能体平台，从传统的单线程、人工切换上下文的 Claw 架构，演进到**多 Agent 并行 + 自动化编排协作**的新一代架构。

### 核心愿景

- **从"单线程"到"并行协作"**：不再是一个 Agent 顺序执行任务，而是多个 specialized Agent 同时工作
- **从"人工切换"到"自动化编排"**：由系统智能地分解任务、分配工作、协调协作，无需人工干预
- **从"单一能力"到"专业分工"**：不同 Agent 具备不同专长（代码专家、文档专家、测试专家等），协同完成复杂任务
- **从"被动执行"到"主动规划"**：系统能自主理解目标、制定计划、监控进度、调整策略

### 为什么需要多 Agent 架构？

传统 Claw 的局限性：
1. **单线程瓶颈**：一个 Agent 同时只能做一件事，效率低下
2. **注意力分散**：同一个 Agent 需要在不同任务间切换，质量下降
3. **能力局限**：单个模型难以在所有领域都达到专家水平
4. **人工依赖**：需要用户不断引导和切换上下文

MineClaw 的突破：
1. **并行执行**：多个 Agent 同时工作，显著提升效率
2. **专业分工**：每个 Agent 专注特定领域，质量更高
3. **智能编排**：系统自动协调，无需人工干预
4. **弹性扩展**：可根据需要动态增加或减少 Agent

## 核心架构设计

### 整体协作模式

MineClaw 采用**混合协作模式**，结合两种核心协作方式：

#### 1. 集群并行模式
多个 Agent 同时工作，各自承担不同子任务，适用于：
- **创意脑洞**：多个 Agent 从不同角度发散思维
- **API 查询与学习**：并行查询文档源、交叉验证
- **大项目代码分工**：约定接口后，每个 Agent 实现一部分
- **多进程测试**：并行执行测试用例，大幅缩短时间

#### 2. 接力转交模式
任务被拆分为多个阶段，每个阶段由专门的 Agent 完成，完成后通过 JSON 工单转交下一 Agent，适用于：
- **持续犯错**：当前 Agent 陷入困境时，换新鲜视角
- **复杂项目分阶段**：每个阶段刷新上下文，避免过长

### 层级化总控架构

```
主总控 (Master Orchestrator) - 最顶层，唯一
    │
    ├── 子总控 1 (Sub-Orchestrator, Level 1)
    │   ├── 可以创建：普通 Agent
    │   ├── 可以创建：更小的子总控 (Level 2，如需要)
    │   └── 可以创建：集群（并行执行多个 Agent）
    │
    ├── 子总控 2 (Sub-Orchestrator, Level 1)
    │   └── ...
    │
    └── 上下文管理 Agent (Context Manager Agent, CMA)
            ↓
       独立监控所有 Agent
```

#### Agent 类型定义

**主总控 (Master Orchestrator)**
- 最顶层的唯一总控
- 负责初始路由决策
- 管理所有子总控的生命周期

**子总控 (Sub-Orchestrator)**
- 可以嵌套，知道自己的嵌套深度 (Level N)
- 可以创建普通 Agent、更小的子总控或集群
- 完成任务后写工单交回给父总控
- 提示词会软性劝阻过深嵌套，但不硬性限制

**普通执行 Agent (Worker Agent)**
- 具体执行任务的无状态执行者
- 可以主动向 CMA 发送求助工单
- 不知道其他 Agent 的存在

**上下文管理 Agent (Context Manager Agent, CMA)**
- 独立监控所有 Agent
- 接收 Agent 求助和处理上下文满载
- 决定回退和转交，但不直接执行

#### 总控 Agent 的特殊能力
- **最强模型配置**：通常使用能力最强的模型
- **创建 Agent 实例**：可以编写提示词动态创建新 Agent
- **编写工作流**：定义任务分解和执行流程
- **计划与评审**：只在每一小步的开始和结束时介入
- **可被重置**：总控犯错也会被上下文管理 Agent 识别并重置

#### 汇报机制
- **分支总控**：处理分支内的反馈
- **主总控**：只在分支结束时介入
- **汇报内容**：只包含完成内容和未解决的问题

### 上下文管理 Agent (Context Manager Agent, CMA)

上下文管理 Agent 是整个工作流的"监军"，独立监控所有 Agent。

#### 触发机制（只有两种情况）

**触发点 1：普通 Agent 主动求助（Worker → CMA）**
- 普通 Agent 发现自己持续犯错 → 生成"求助工单"
- 求助工单直接发送给 CMA（不是主总控！）
- 求助工单内容：
  - 我是谁（Agent ID/角色）
  - 我在做什么任务
  - 我尝试了什么，失败了多少次
  - 相关的错误信息
  - 我建议回退到哪个 Checkpoint（可选）

**触发点 2：上下文满载（唯一的自动触发点）**
- 检测到：当前上下文长度 ≥ 阈值（如 90%）
- 自动触发：CMA 进行清理
- 其他情况不自动触发，因为每次输入的上下文通常很长

#### CMA 干预流程

**情况 A：收到 Agent 的主动求助**
1. CMA 接收求助工单 → 分析情况是否确实严重
2. CMA 决定应该回退到哪个 Checkpoint
3. CMA 触发当前层级的总控进行转交
4. 总控负责实际的回退和创建新 Agent
- （CMA 是"监军"，不是"主帅"，不直接操作执行）

**情况 B：上下文满载**
1. CMA 执行上下文清理（裁剪）
   - 分析哪些可以裁剪（反复失败的尝试、冗余对话等）
   - 执行裁剪（像修改文件一样编辑上下文）
   - 创建 Checkpoint（保存裁剪前的状态）
2. CMA 判断："这个 Agent 是在持续犯错吗？"
   - 是 → 触发回退 + 转交流程（同情况 A）
   - 否 → 清理完毕，让任务继续执行

#### 裁剪策略参考（提示词模板）
- **持续犯错部分**：裁剪掉反复失败的尝试
- **Bug 解决过程**：保留结果，裁剪过程（未来用于经验学习）
- **终端长输出**：裁剪冗余，如测试日志只保留测试成功/失败总结
- **冗余对话**：删除重复、无关的内容

#### 裁剪质量评估
- 裁剪提示词会被记录
- 由评审 Agent 或用户根据任务完成质量评分
- 评分结果将参与未来的提示词自动优化（Phase 7+）

### 工单机制 (Work Order)

接力转交时使用 JSON 格式工单，包含：

```
工单内容规范：
├── 已完成部分详细标注
├── 相关文件的相对路径列表
└── 下一阶段的详细计划
```

**设计原则**：工单应简洁、聚焦，仅包含必要信息，避免成为新的上下文负担。

### 路由模型 (Router Model)

#### 触发时机
1. **任务开始时**：初始判断使用单 Agent 还是多 Agent 模式
2. **上下文裁剪后**：上下文被管理 Agent 裁剪后，重新评估路由策略

#### 路由决策逻辑

**简单任务** → 单总控 Agent 模式（或创建单个普通 Agent）

**复杂任务** → 判断使用哪种子总控模式

---

### 复杂任务的两种执行模式

#### 模式 A：嵌套子总控接力模式
**适用场景**：复杂任务，各步骤之间强依赖顺序，每一步都很复杂

**执行流程**：
```
主总控 → 创建子总控 1 (Level 1)
    ↓
子总控 1 执行（内部可能创建集群并行执行）
    ↓
子总控 1 完成 → 写工单 → 主总控
    ↓
主总控更新计划 → 创建子总控 2 (Level 1)
    ↓
子总控 2 执行 → 写工单 → 主总控
    ↓
... 重复直到所有阶段完成 ...
    ↓
主总控汇总结果
```

**示例**：主总控 → (总控1→集群→总控1写工单→主总控) → (主总控更新计划后创建新总控2→集群)...

---

#### 模式 B：平行子总控集群模式
**适用场景**：复杂任务，但各部分可以分别独立执行

**执行流程**：
```
主总控 → 分解任务为多个独立子任务
    ↓
主总控同时创建：
    ├─ 子总控 1 (Level 1) - 负责子任务 A
    ├─ 子总控 2 (Level 1) - 负责子任务 B
    └─ 子总控 3 (Level 1) - 负责子任务 C
    ↓
所有子总控并行执行（每个内部可能创建自己的集群或 Agent）
    ↓
子总控 1 完成 → 写工单 → 主总控
子总控 2 完成 → 写工单 → 主总控
子总控 3 完成 → 写工单 → 主总控
    ↓
主总控汇总所有结果
```

**示例**：主总控 → (总控1, 总控2... 计划后创建 → Agent/集群 Agents)...

---

**混合模式**：一个项目可以灵活组合使用两种模式
- 集群为每个 Agent 分派任务
- 每个 Agent 自己把任务拆分成 Baby Steps，通过接力转交完成
- 每个集群分支都有独立的分支总控管理

### 工具掩码策略

#### 掩码范围
- **统一处理**：MCP 工具和本地工具统一管理
- **掩码单位**：单个工具（而非整个 MCP 服务器）

#### 工具权限分类
- **MCP 工具**：只有可用/不可用两种状态（权限由 MCP 服务器端控制）
- **本地工具**：支持只读/读写一键切换，也可单个工具精细配置
- **终端工具**：所有 Agent 都可以使用，无限制

#### 分配策略
- **主总控 Agent**：拥有全部权限
- **Worker Agent**：由总控在分配任务时动态配置工具权限
- 确保分配的工具通常足够完成任务
- 避免工具过多导致的分心

### Checkpoint 与会话管理

- **强关联设计**：Checkpoint 与 Session 一一对应
- **生命周期管理**：
  - Session 归档 → 相关 Checkpoint 归档或删除
  - 由 AgentFS 提供完整实现
- **轻量设计**：完全不会成为系统负担

### 转交预热 (Handover Warm-up)

**必须流程**：在正式转交任务前，需要进行预热

- 让新 Agent 先理解工单内容
- 确认关键信息无遗漏
- 准备好后再正式接管

### 线性流水线设计

- **设计原则**：流水线是线性的，不支持Agent间通信
- **消息追踪**：每个 Agent 的回复都有唯一的消息 ID 记录
- **简化协作**：避免复杂的Agent间依赖，让流程更清晰可控

### 独立大屏与全景可视化前端

MineClaw 放弃了与现有 IDE 强绑定的插件形式，而是选择成为一个拥有独立控制台的“AI 自动化工厂”。

#### 为什么选择独立前端？
- **全景工作流可视化**：可以直观展示 Agent 树状层级（谁在指挥谁）、工单 (Work Order) 的流转路径。
- **Checkpoint 交互式管理**：像 Git 树一样查看所有快照，实现时间轴式的代码变更对比 (Diff) 与一键回退。
- **零干预模式 (Autonomous Mode)**：用户只需看大盘监控，出现异常时（如 CMA 告警或 Agent 主动求助）再进行轻量级干预。
- **掌控底层存储**：保留 Rust 后端对 `AgentFS` 和 Checkpoint 的绝对控制权，不再受限于第三方 IDE 的客户端沙箱体系。

#### 集成策略
- **全异步监控 API**：基于 REST + SSE/WebSocket 打造实时状态总线。
- **Flutter 跨平台控制台**：统一的控制中心，负责任务下发、进度追踪和人工审批。

## 关键设计决策回顾

### 1. 混合协作模式
- 结合集群并行和接力转交两种模式的优势
- 一个项目可以灵活组合使用两种模式

### 2. 层级化总控 + 可嵌套子总控
- 主总控 + 子总控的多层架构
- 子总控可以嵌套，知道自己的嵌套深度 (Level N)
- 提示词软性劝阻过深嵌套，但不硬性限制（保持灵活性）
- 子总控完成任务后写工单交回给父总控
- 总控只在关键节点介入，避免过度干预
- 总控也可以被重置，避免单点故障

### 3. 复杂任务的两种执行模式
- **嵌套子总控接力模式**：适用于强依赖顺序的复杂任务
- **平行子总控集群模式**：适用于可并行分解的复杂任务
- 主总控根据任务性质选择合适的模式

### 4. 上下文管理 Agent (CMA) 是"监军"
- 独立监控所有 Agent，不直接执行
- 只有两个触发点：Agent 主动求助、上下文满载
- 收到求助后决定回退和转交，通知总控执行
- 上下文满载时清理，并判断是否在持续犯错

### 5. Agent 可以主动求助
- 普通 Agent 发现持续犯错时，生成求助工单发给 CMA
- 求助工单包含：犯错情况、建议回退的 Checkpoint
- CMA 而不是主总控接收求助

### 6. 工单机制简洁聚焦
- 只包含必要信息：已完成、文件路径、下一步计划
- 避免成为新的上下文负担

### 7. 工具掩码精细控制
- 按单个工具粒度掩码
- 终端工具全开放
- 总控负责为 Agent 分配合适的工具

### 8. Checkpoint 与会话强关联
- 简化管理，避免孤立 Checkpoint
- 由 AgentFS 提供支持，轻量高效
- CMA 回退时使用 Checkpoint

### 9. 转交预热必须
- 确保平滑过渡
- 避免信息丢失

### 10. 线性流水线
- 简化协作逻辑
- 避免复杂的Agent间依赖
- 完整的消息 ID 追踪

### 11. 裁剪质量可评估
- 为未来的自动优化积累数据
- 支持人工和自动评审

### 12. 为经验学习预留接口
- Phase 6 就开始为 Phase 7 准备数据结构
- 确保平滑过渡到持续进化阶段

### 13. 独立控制台优先 (取代 ACP)
- 放弃 ACP 的原因在于其“IDE 主导”的哲学与 MineClaw “后端自治”的 Checkpoint/AgentFS 机制冲突。
- 采用独立大屏前端可以实现对复杂多 Agent 协作和工单流转的完美可视化。
- 提供 API-First 的设计，通过 SSE/WebSocket 保障前端对底层状态的实时掌控。

## 开发原则

1. **Baby Steps™**: 每次只做一个小的、有意义的改变
2. **The Process is the Product**: 过程和结果同样重要
3. **Incremental Validation**: 每一步都要验证
4. **Documentation First**: 先写文档再实现
5. **Concurrency Safety**: 并发安全是首要考虑
6. **Fault Tolerance**: 设计要考虑容错和恢复
7. **线性流程**: 保持流水线线性，简化协作
8. **可观测性**: 所有关键操作都要可追踪、可审计
